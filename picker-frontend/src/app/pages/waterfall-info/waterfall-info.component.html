WATERFALL.

The waterfall model is one of the most widely used methods, especially in software development. It belongs to the sequential traditional models. The model has different phases, which must be run off in a certain order. Thereby, a new phase can only be started when the previous phase has been completed. The following phases are observed in the model:

1. Requirements Engineering
2. Design and Implementation
3. Testing
4. Release
5. Maintenance

The first phase of the waterfall model is the requirements engineering phase. In this phase, the requirements for the product are described in detail. The requirements of the customers and, if applicable, other stakeholders are recorded and documented. Fixed deadlines are defined and the resources and costs for implementing the project are estimated. In the second phase, the system to be created is first designed and then implemented. Especially in software development, the tools and technologies used are bundled in the design phase and an architecture of the system is designed. In the third phase, the system is tested to ensure that it is ready for release. All functional requirements are tested. All non-functional requirements in software projects, which the customer has requested depending on the use case, are also tested for their appropriateness and quality. In the Release phase, the solution is accepted by the customer. The system is transferred to production and the customer is provided with the relevant documentation and instructions. The last phase speaks of a maintenance phase. Depending on what agreements have been made with the customer, the responsibility for maintenance may lie in the hands of the developers. In this case, the system is maintained on request and optimized if necessary.

Pros:

  • Fixed time sequences with clear responsibilities are established
  • The time and budget are conscientiously estimated in the planning at the beginning
  • If all requirements are there and no changes are expected, then development becomes easier
  • The model can be easily used due to its linearity
  • The model is used for many projects and the structure is therefore known

Cons:

  • Risk of incorrect planning due to the increasing complexity in the demands of e.g.: Customers
  • It is not always possible to define all requirements directly at the beginning in the planning phase
  • Error handling in the middle of the project is usually very costly
  • Due to the importance of the planning phase, a great deal of time is invested in this phase
  • Changing customer requirements in the middle of the project can usually not be taken into account
  • Signing off on organizational matters at each stage leads to increased effort and expense
  • Problems are postponed to the next phase so that the time schedule is adhered to
  • Customer feedback is obtained only at the end of the project
  • A finished software is provided only at the end of the project without previous prototypes
